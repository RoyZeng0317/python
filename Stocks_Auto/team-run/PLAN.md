# PLAN

## Restatement & Success Criteria

**使用者想要的東西**：在 `Stocks_Auto/` 目錄下建立一支（或一組）自動化腳本，每天：
1. 從網路上抓取股票相關新聞（目前草稿來源：Yahoo奇摩股市首頁 + 一支 YouTube 影片在台北時間 09:00–13:00 區間的內容）。
2. 對抓到的新聞做「總整理」（去重、分類、摘要、產出報告）。
3. **總整理/摘要邏輯明確不可使用現成 LLM/AI 模型**（不可呼叫 GPT/Claude/Gemini 等 API），必須用傳統演算法（RSS/HTML 爬蟲、去重、TF-IDF、抽取式摘要 TextRank、規則式分類/樣板）完成，理由是避免消耗 token/API 費用。
4. 輸出成 `yyyy-mm-dd.json`（沿用草稿命名慣例），目前草稿寫死的輸出路徑 `C:\Users\roy\Documents\GitHub\Websites\Stocks\backend\data\` 是另一台機器的路徑，需要修正。
5. 草稿預留 `rules.md`（篩選規則，目前只有第1條完整）與 01_Data_Search.py 內的「資料分析」區塊（總整理邏輯應該掛在這裡或独立模組）。
6. `10_LangChain.py`、`11_LamaIndex.py`、`12_Martix_DataSheet.py` 三個空檔案是先前考慮用 LLM 框架的殘留，因使用者已明確排除 LLM 路線，需要重新決定去留與用途。

**Success Criteria（完成的定義）**：
- 有一份清楚、使用者已對所有開放問題拍板的規劃文件（本檔）與後續研究/實作文件。
- 最終腳本（由 Builder 產出）能夠：從已核准的來源抓新聞 → 套用 rules.md 篩選 → 去重 → 依股票/公司分組 → 用非LLM演算法產出總整理 → 輸出 JSON（+使用者選定的人類可讀格式）到 repo 內正確路徑。
- 全程程式碼中**不出現任何 LLM API 呼叫**（可用 import/依賴掃描驗證：不得出現 openai、anthropic、google.generativeai 等套件）。
- 使用者能手動執行一次，全流程不崩潰、產出檔案格式符合預期。

**信心 [High]**：以上restatement直接對應使用者原話與現有草稿內容，無需額外推測。

## Approach & Verification per Sub-task

### M6 摘要/彙整演算法（核心決策，附三選項比較）

| 方案 | 說明 | 優點 | 缺點 |
|---|---|---|---|
| **A. TextRank/LexRank 抽取式摘要** | 將文章句子建圖（以句子間 TF-IDF 向量的 cosine similarity 為邊權重），跑類PageRank演算法選出代表性句子 | 不需訓練、無需外部API、可自動抓出文章中「多句共識」的重點句 | 中文需先斷詞（jieba/CKIP），實作與依賴較複雜；抽出的句子可能失去上下文變得不通順；對短文章效果有限 |
| **B. 規則式「標題+首段」擷取** | 直接取新聞標題 + 首段（或前100-150字），利用中文財經新聞「倒金字塔」寫法（重點通常在前面）| 極簡單、零額外NLP依賴、穩定可預期、最符合「總整理」= 快速瀏覽的使用情境 | 本質上不是「壓縮摘要」而是「擷取」；若來源文章不遵守倒金字塔寫法，品質會打折 |
| **C. TF-IDF 關鍵詞抽取/聚類** | 對當日所有文章算 TF-IDF，抽每篇/每組的高權重詞作為關鍵字標籤，可再依關鍵字做新聞聚類 | 計算快、可用於「熱門關鍵字」呈現、可輔助分組(M5) | 不是文章級摘要，只能作輔助資訊，不能單獨當「總整理」的全部內容 |

**建議 [High confidence]，理由：使用者要的是「總整理」而非深度壓縮式摘要，且明確要避免額外NLP依賴的維護成本** → v1 採 **方案B（標題+首段）為主要摘要內容 + 方案C（TF-IDF關鍵字標籤）為輔助**，兩者皆不需中文斷詞的複雜依賴即可先做（TF-IDF可用簡單n-gram或jieba基礎分詞）。方案A（TextRank）列為 v2 選配，待 Q10 若選 (b) 或使用者反饋 v1 資訊量不足時再加入，屆時需額外引入 jieba 斷詞依賴。

**驗證方法**：對一天實際抓到的新聞樣本（約10-20篇）跑過模組，人工檢視輸出摘要是否可讀、資訊是否失真；並用 `grep -ri "openai\|anthropic\|generativeai\|import.*gpt"` 掃描全部程式碼確認零LLM依賴。

**Backtrack trigger**：若人工檢視發現方案B摘要在多數樣本上無法呈現文章重點（例如來源網站的首段常是編輯套話而非內容重點），則改評估導入方案A。

### M2 資料擷取器

### M4 去重模組
